== Das Problem

Wenn eine Klasse ihre Abhängigkeiten selbst erzeugt, entsteht enge Kopplung:

[source,typescript]
----
class OrderProcessor {
  process(order: Order): void {
    const validator = new OrderValidator()  // <1>
    const emailer = new EmailService()      // <2>
    validator.validate(order)
    emailer.sendConfirmation(order)
  }
}
----
<1> Harte Abhängigkeit — nicht testbar
<2> Versteckte Abhängigkeit — nicht sichtbar in der API

== Die Lösung

Abhängigkeiten durch den Konstruktor injizieren:

[source,typescript]
----
class OrderProcessor {
  constructor(
    private validator: OrderValidator,
    private emailer: EmailService
  ) {}

  process(order: Order): void {
    this.validator.validate(order)
    this.emailer.sendConfirmation(order)
  }
}
----

== Warum das wichtig ist

[cols="1,1"]
|===
| Aspekt | Vorteil

| Testbarkeit
| Mocks/Stubs statt echter Services

| Austauschbarkeit
| Implementierung wechseln ohne Consumer zu ändern

| Lesbarkeit
| Alle Abhängigkeiten im Konstruktor sichtbar

| Wiederverwendbarkeit
| Klasse funktioniert mit verschiedenen Implementierungen
|===

== Faustregel

Wenn du `new SomeService()` _innerhalb_ einer Methode schreibst, hast du wahrscheinlich ein Design-Problem. Extract to constructor.
